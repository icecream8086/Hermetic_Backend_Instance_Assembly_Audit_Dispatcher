# 复现提示词

通用三步法，适用于任何代码库审计+重构任务。

---

## 第一步：形式化验证

把代码和设计文档对照，找出差异。

```
对比 <A模块> 和 <B模块> 的差异。理论上 B 是 A 的子集。列出路由/方法/权限矩阵，找出不一致的地方。
```

```
去 <SPEC文件夹> 核对设计文档，形式化验证当前实现是否符合规格。列出差异表：文档规定 vs 代码实现。
```

```
读取 <设计文档目录> 的全部内容，提取架构意图和形式化模型，与当前代码逐条对照。
```

---

## 第二步：静态分析+审计

用 ESLint 和 grep 扫描，锁定违规和死代码。

```
审核现有更改。严打绕过 ESLint 的投机取巧行为（as any、as unknown as、函数参数 any、泛型 any）。
符合文档即放行，输出每条违规的行号+规则+严重度。
```

```
改进 ESLint 阻止此类行为。检查是否有运行时逃逸舱口（如 z.any()）绕过类型检查，
若有则加 no-restricted-syntax AST 选择器封堵。
```

```
用静态分析扫描确认：哪些函数仅被废弃路径调用、哪些类未被 import、哪些构造参数从未使用。
输出完整死代码清单（文件名+行号+行数）。
```

---

## 第三步：架构决策+重构

做出取舍，一次性铲干净。

```
<A模块> 是正版 <K8s API 全集>，<B模块> 是简化版 <包装器>。
目标：补全 A 的所有能力，B 只做透传+叠加薄层。
给出实现计划和变更清单（文件+行数+操作）。
```

```
有没有激进一点不留后路的改进方案？要求零死代码、零过渡层、零 fallback 分支。
```

```
假设 <A> 是完整类 <K8s API>，<B> 是简化版，铲掉所有过渡代码还要做什么？
列出需删除的所有函数/类/分支/构造参数，和需补全的所有方法/路由。净变更行数。
```

---

## 每步后验证

```
npx eslint <改过的文件>  → 0 新增错误
npx tsc --noEmit          → 0 错误
npx vitest run            → 全量通过
git diff --stat           → 确认变更范围
```
